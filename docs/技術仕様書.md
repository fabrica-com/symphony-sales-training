# Symphony 営業研修プログラム 技術仕様書

本ドキュメントは、このリポジトリを再現するために必要な技術仕様をまとめたものです。**管理機能（app/(admin)/）は対象外**とし、一般ユーザー向けの研修・学習機能に限定して記載します。

---

## 1. 概要

### 1.1 アプリケーションの目的

- **Symphony 研修プログラム**：新卒からマネージャーまで、営業力強化のための体系的ミニ研修（全100本想定）を提供する Web アプリケーション
- カテゴリ別カリキュラム、インタラクティブセッション、総合テスト、Deep Dive 読み物、学習進捗ダッシュボードを提供

### 1.2 主な機能

| 機能 | 説明 |
|------|------|
| 認証 | Google OAuth によるログイン／ログアウト |
| トップ／カリキュラム | カテゴリ一覧表示、カテゴリ詳細・研修一覧 |
| 研修詳細 | 研修概要・目的・セクション・名言・まとめ・Deep Dive リンク |
| 研修セッション | チェックイン→復習→ストーリー→インフォグラフィック→クイックチェック→シミュレーション→振り返り→完了（DB の session_contents に依存） |
| 総合テスト | カテゴリ別テスト（問題はローカル `lib/test-data.ts` または DB） |
| Deep Dive | カテゴリ C 章・研修別の深掘り読み物（DB またはローカル `lib/deep-dive-content.ts`） |
| ダッシュボード | 累計ポイント・受講済み数・学習時間・平均スコア、カテゴリ別進捗・総合テスト結果・受講履歴 |
| プロンプト編集 | 研修ごとのロープレ・評価用プロンプト編集（AI 生成対応） |

## 4. 認証・セッション

### 4.1 認証方式

- **Google OAuth 2.0**（Supabase Auth）
- ログイン：`/login` → `loginWithGoogle()` → Supabase `signInWithOAuth({ provider: "google" })` → リダイレクト先 `{origin}/auth/callback`
- コールバック：`/auth/callback`（GET）で `code` を `exchangeCodeForSession` し、成功時は `next` クエリまたは `/dashboard` へリダイレクト

### 4.2 プロキシ（認証ガード）

- **Next.js 16** では従来の `middleware` が **proxy** に相当
- **ファイル**：ルートの `proxy.ts`
- **処理**：`lib/supabase/proxy.ts` の `updateSession(request)` でクッキーからセッション更新＋`getUser()` 取得
- **公開パス**：`/login`, `/auth/callback`, `/api/*`, `/_next/*`, `favicon.ico`, 静的画像（`.svg`, `.png` 等）
- **未ログイン**で上記以外にアクセス → `/login` へリダイレクト（`redirect` クエリで元 URL を保持可能）
- **ログイン済み**で `/login` にアクセス → `/dashboard` へリダイレクト
- **matcher**：`((?!_next/static|_next/image|favicon.ico|.*\\.(?:svg|png|jpg|jpeg|gif|webp)$).*)`

### 4.3 Supabase クライアント

| 用途 | ファイル | 説明 |
|------|----------|------|
| ブラウザ | `lib/supabase/client.ts` | `createBrowserClient`（@supabase/ssr）。都度生成しキャッシュしない |
| サーバー | `lib/supabase/server.ts` | `createServerClient`（next/headers の cookies）。リクエストごとに生成 |
| サーバー（特権） | `lib/supabase/admin.ts` | `createClient`（SERVICE_ROLE_KEY）。シングルトンキャッシュ。RLS をバイパス |

### 4.4 プロファイル

- **テーブル**：`public.profiles`
- **トリガー**：`auth.users` への INSERT 時に `handle_new_user()` で `profiles` に 1 行挿入（OAuth の `full_name`/`name` を利用）
- **主な列**：`id`（auth.users.id）, `email`, `name`, `department`, `join_date`, `role`（employee / manager / admin）

### 4.5 クライアント側の認証コンテキスト

- **ファイル**：`lib/auth-context.tsx`
- **提供**：`AuthProvider` でラップし、`useAuth()` で利用
- **保持**：`user`（id, email, name, department, joinDate, role）, `trainingLogs`, `userProgress`, `testResults`, `isLoading`
- **操作**：`loginWithGoogle`, `logout`, `addTrainingLog`, `addTestResult`, および各種 getter（`getTestResultsByCategory`, `getBestTestResult`, `getTotalPoints`, `getCompletedTrainings`, `getProgressPercentage`）
- **データ取得**：サインイン／初期表示時に `loadUserDataFromServer(userId)` を呼び、`training_sessions` / `user_training_progress` / `category_test_results` を取得して state にマッピング

### 4.6 認証必須ページ

- `lib/with-auth.tsx` の `withAuth` HOC でラップするページは、未ログイン時に `/login` へリダイレクト
- 実際の認証ガードは **proxy** で行い、ダッシュボード等は未ログインなら proxy で `/login` に飛ばされる

## 5. データベース（Supabase / PostgreSQL）

### 5.1 マイグレーション

- **場所**：`supabase/migrations/`
- **適用順の例**：
  1. `20231231000000_create_profiles_table.sql` — profiles
  2. `20240101000000_create_session_contents.sql` — session_contents
  3. `20240101000001_create_training_tables.sql` — training_categories, trainings, deep_dive_contents, category_tests, category_test_questions
  4. その他（quiz 更新、カテゴリ C テスト、プロファイルトリガー等）
  5. `20260130115445_remote_schema.sql` — training_sessions, user_training_progress, category_test_results 等のリモート反映

### 5.2 主要テーブル（再現に必要な概念）

| テーブル | 用途 |
|----------|------|
| `profiles` | ユーザー拡張情報（名前・部署・役割等） |
| `training_categories` | カテゴリ（id, name, description, total_duration, target_level, color, display_order） |
| `trainings` | 研修（id, category_id, title, subtitle, duration, level, detail(JSONB), display_order） |
| `session_contents` | 研修セッション用コンテンツ（training_id 単位で 1 行。badge, mood_options, review_quiz, story, infographic, quick_check, quote, simulation, reflection, action_options, work, deep_dive 等を JSONB） |
| `deep_dive_contents` | Deep Dive 読み物（training_id, title, subtitle, introduction, sections(JSONB), conclusion, reference_list） |
| `category_tests` | カテゴリテスト設定（category_id, category_name, total_questions, passing_score, time_limit） |
| `category_test_questions` | テスト問題（category_test_id, question_number, question, options(JSONB), correct_answer, explanation, source） |
| `training_sessions` | 受講記録（user_id, training_id, category_id, training_title, attempt_number, duration_seconds, overall_score, max_score, completed_at, category_name 等） |
| `user_training_progress` | ユーザー別研修進捗（user_id, training_id, category_id, status, completed_at 等）。一意制約 (user_id, training_id) |
| `category_test_results` | 総合テスト結果（user_id, category_id, category_name, score, percentage, passed, correct_count, total_questions, duration, attempt_number 等） |

- RLS が有効なテーブルは、ポリシーに従い「読取は公開または認証ユーザー」「書込は認証または service_role」などで制御
- 受講記録・テスト結果の**挿入**は Server Actions から **createAdminClient()** を使用（SERVICE_ROLE_KEY）

### 5.3 アプリ側の型・データ取得

- **DB 用型**：`lib/db/types.ts`（`DbCategory`, `CategoryTest`, `CategoryTestQuestion` 等）
- **カテゴリ・研修・Deep Dive・カテゴリテスト**：`lib/db/categories.ts`（`getAllCategories`, `getCategoryByIdFromDb`, `getTrainingByIdFromDb`, `getDeepDiveContentFromDb`, `getCategoryTestFromDb` 等）
- **セッションコンテンツ**：`lib/session-data/db.ts`（`getSessionContentFromDB`）、型は `lib/session-data/types.ts`（`SessionContent` 等）
- **静的フォールバック**：カテゴリ・研修メタは `lib/training-data.ts`（`categories`, `getTrainingById`）、総合テスト問題は `lib/test-data.ts`（`getCategoryTest`, `calculateTestScore`）、Deep Dive は `lib/deep-dive-content.ts`（`getDeepDiveContent`）でローカル定義を併用可能

## 6. ルーティング構成（admin 除く）

| パス | 種別 | 説明 |
|------|------|------|
| `/` | ページ | トップ。カテゴリ一覧（`getAllCategories()`）＋ヒーロー＋フッター |
| `/login` | ページ | ログイン（Google）。エラー時はクエリでメッセージ表示 |
| `/auth/callback` | GET | OAuth コールバック。code 交換後リダイレクト |
| `/dashboard` | ページ | ダッシュボード（累計 pt・受講数・学習時間・平均スコア、タブ：進捗・総合テスト・履歴） |
| `/category/[id]` | ページ | カテゴリ詳細。`getCategoryByIdFromDb(id)`＋研修一覧。C のとき Deep Dive リンク |
| `/category/[id]/test` | ページ | 総合テスト。`getCategoryByIdAction` / `getCategoryTestAction`（ローカルテストデータ併用） |
| `/category/C/deep-dive` | ページ | カテゴリ C の Deep Dive（固定） |
| `/training/[id]` | ページ | 研修詳細。`getTrainingByIdFromDb`＋`getSessionContentFromDB`＋Deep Dive 有無。サイドに「研修開始」「プロンプト編集」 |
| `/training/[id]/session` | ページ | 研修セッション。`getTrainingById`（静的）＋`getSessionContentFromDB`。無い場合は「準備中」 |
| `/training/[id]/prompts` | ページ | プロンプト編集（ロープレ・評価用。AI 生成あり） |
| `/training/[id]/deep-dive` | ページ | 研修別 Deep Dive |
| `/api/category-test/[id]` | API | カテゴリテスト用（必要に応じて参照） |

- 動的ルートは `params` を `Promise<{ id: string }>` で受け取り、`await params` で利用
- `export const dynamic = "force-dynamic"` を指定しているページは DB 依存のため静的生成しない

## 10. 再現の手順（要点）

1. **リポジトリのクローン**
   - 管理機能 `app/(admin)/` は含めずに再現する場合は、そのディレクトリを参照しないか、ビルド対象から外す。

2. **Node とパッケージ**
   - Node.js LTS を用意し、`npm install`（または `bun install`）で依存関係を導入。

3. **環境変数**
   - `.env.local` に `NEXT_PUBLIC_SUPABASE_URL`, `NEXT_PUBLIC_SUPABASE_ANON_KEY`, `SUPABASE_SERVICE_ROLE_KEY` を設定。

4. **Supabase プロジェクト**
   - プロジェクト作成後、Auth で Google プロバイダーを有効化し、リダイレクト URL に `https://<origin>/auth/callback` を登録。
   - 上記マイグレーションを順に適用し、必要ならシードで `training_categories`, `trainings`, `session_contents`, `deep_dive_contents`, `category_tests`, `category_test_questions` を投入。

5. **開発・ビルド**
   - `npm run dev` で開発サーバー起動。
   - `npm run build` → `npm run start` で本番ビルド確認。
   - トップ `/`、`/login`、`/dashboard`、`/category/[id]`、`/training/[id]`、`/training/[id]/session` 等の遷移と、ログイン・受講記録・テスト結果の保存が行えることを確認。

6. **注意**
   - `training-data.ts` / `test-data.ts` / `deep-dive-content.ts` は静的データとして参照されるため、DB にデータがない場合でも一部画面は表示可能。セッションコンテンツは `session_contents` に依存するため、未登録の研修は「準備中」となる。
   - ダッシュボードの `userProgress` は `Record<number, UserProgress>` であり、`userProgress.size` のように扱っている箇所は、実際には `Object.keys(userProgress).length` などで件数取得する必要がある（実装時要確認）。
